# 三层算力架构

> **免责声明**：基于公开传闻与产业链推演，**非苹果官方架构文档**。

---

## 模块概述

N50 采用「眼镜采集 + iPhone 推理 + 云端按需」的三层算力模型，是苹果在**重量/续航/AI 能力**三角约束下的工程取舍。与 Meta Ray-Ban 将更多 AI 压在镜端 AR1 不同，苹果选择把 Neural Engine 算力集中在已量产的 iPhone 上。

---

## N50 传闻规格

### 第一层：眼镜端（Edge-Lite）

| 任务 | 说明 |
|------|------|
| 传感器采集 | 双摄像头、3–4 颗 MEMS 麦克风 |
| 轻算力 | 语音唤醒、ISP 预处理、端侧预筛选、音频降噪 |
| SoC | N401，Watch S 系魔改，功耗费 <2W |
| 不做 | 大模型推理、复杂 CV、独立联网 |

### 第二层：iPhone 端（Edge-Primary）

| 任务 | 说明 |
|------|------|
| 主 AI 推理 | Visual Intelligence、识物/翻译/导航语义 |
| 芯片 | A18/A19 系列 16 核 Neural Engine，~35 TOPS |
| 系统 | iOS 27 可穿戴框架、升级版 Siri |
| 连接 | 蓝牙 LE + 传闻 UWB 传特征/低延迟控制信令 |

### 第三层：云端（Cloud-On-Demand）

| 任务 | 说明 |
|------|------|
| 触发条件 | 复杂问答、长上下文、部分翻译/搜索 |
| 基础设施 | Private Cloud Compute、Apple 数据中心 |
| 原则 | 非默认路径；隐私敏感视觉数据优先本地 |

## 供应链或系统逻辑

```
┌──────────────────────────────────────────────────────┐
│ 眼镜层：看/听 → ISP/唤醒/预筛选 → BT/UWB 上行        │
│         延迟目标：采集→传输 <50ms（传闻）             │
├──────────────────────────────────────────────────────┤
│ iPhone层：NE 推理 → Siri/AI 响应 → 音频回传眼镜      │
│         延迟目标：端到端 <150–200ms（行业推演）       │
├──────────────────────────────────────────────────────┤
│ 云端层：按需大模型 → 结果经 iPhone 聚合 → 语音输出   │
└──────────────────────────────────────────────────────┘
```

**数据流策略（推演）**：
1. 眼镜上传**压缩特征/缩略图**而非原始 4K 流（省带宽与隐私）
2. iPhone 本地 NE 完成 80%+ 视觉 AI 任务
3. 仅当本地模型置信度不足或用户显式请求时上云

## 优势分析

| 对比维度 | N50 三层架构 | 镜端一体化（Meta AR1） |
|---------|-------------|----------------------|
| 眼镜功耗 | 极低（<2W） | 中等（AR1 全功能 SoC） |
| AI 算力上限 | 随 iPhone 代际升级 | 受 AR1 TDP 锁定 |
| 模型更新 | OTA 更灵活（手机端） | 依赖眼镜固件 |
| 独立使用 | 弱（需 iPhone） | 较强（Wi-Fi/部分离线） |

## 前沿未解问题

- **带宽瓶颈**：双摄像头 + 手势 CV 同时开启时，BT 5.x 是否足够？UWB 传什么粒度数据？
- **热量分布**：推理在 iPhone，长时间 Visual AI 是否导致手机发热降频？
- **多设备仲裁**：同一 iPhone 连眼镜 + Watch + AirPods 时的算力/音频路由
- **离线降级**：隧道/飞行模式下功能清单尚未清晰
